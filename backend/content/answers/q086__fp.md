question: "What are SQL window functions and how do they differ from GROUP BY?"

The fundamental tension: sometimes you need an aggregate *and* the individual rows that produced it, in the same result. `GROUP BY` cannot give you this—it's a **many-to-one collapse**. It partitions rows into groups and destroys the rows, emitting one summary per group. Ask "each employee's salary *and* their department average" and GROUP BY is stuck: the row-level detail is gone.

Window functions resolve the tension by decoupling **aggregation from collapse**. A window function computes over a set of rows (the "window") defined by `OVER (PARTITION BY ... ORDER BY ...)`, but attaches the result *back to every row*—a **many-to-many** mapping that preserves cardinality. Same number of rows in, same out, now enriched with context.

Why is this powerful? Because the window can *slide* and *order*, unlocking a whole class of computations GROUP BY can't express at all:
- **Rankings**: `ROW_NUMBER`, `RANK`, `DENSE_RANK`.
- **Inter-row references**: `LAG`/`LEAD` for period-over-period change.
- **Running totals / moving averages**: a frame that grows or slides with `ORDER BY`.

The deeper principle: GROUP BY answers "what is true of each *group*?"; windows answer "what is true of each *row relative to its neighbors*?" The non-obvious insight—window functions run *after* `WHERE`/`GROUP BY` in the logical query order, which is exactly why you often need a subquery to filter on a window result like `rank = 1`.
