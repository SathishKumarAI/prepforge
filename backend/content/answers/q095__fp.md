The interesting part isn't syntax—it's that "second-highest" is **ambiguous until you define ties**, and each disambiguation maps to a different SQL construct. Nail the semantics and the query writes itself.

**Interpretation A: the second-highest *distinct* value.** If several people earn the top salary, they don't count as first *and* second. You want the second unique salary. The cleanest expression:

```sql
SELECT MAX(salary) FROM employees
WHERE salary < (SELECT MAX(salary) FROM employees);
```

This reads as its own definition: "the largest salary strictly below the largest salary." It also returns `NULL` gracefully when no second value exists—often the desired edge behavior.

**Interpretation B: the second row by rank.** Use a window function and choose the ranking that encodes your tie rule:

```sql
SELECT salary FROM (
  SELECT salary, DENSE_RANK() OVER (ORDER BY salary DESC) AS rnk
  FROM employees
) t WHERE rnk = 2;
```

`DENSE_RANK` gives "second distinct value"; `RANK`/`ROW_NUMBER` would answer differently under ties—which is exactly the point.

The deeper principle: SQL is **declarative**—you specify *what* set you want, and correctness lives entirely in matching your predicate to the real question. The non-obvious insight: `LIMIT 1 OFFSET 1` (the popular one-liner) silently assumes *no duplicate top salaries*; it answers a subtly different question and breaks on ties.
