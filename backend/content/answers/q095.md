The answer the interviewer is really probing is **do you handle duplicates and empty results correctly**, so I lead with the robust version and explain the trap.

Portable subquery approach:

```sql
SELECT MAX(salary) AS second_highest
FROM employees
WHERE salary < (SELECT MAX(salary) FROM employees);
```

This finds the largest salary strictly below the overall max. It naturally handles **tied top salaries** (multiple people earning the max still collapse to one value) and returns `NULL` gracefully when there's no second value — no error, no wrong row.

The modern, more general approach uses a window function, which extends cleanly to "Nth highest":

```sql
SELECT DISTINCT salary
FROM (
  SELECT salary, DENSE_RANK() OVER (ORDER BY salary DESC) AS rnk
  FROM employees
) t
WHERE rnk = 2;
```

`DENSE_RANK` is the deliberate choice: it gives tied salaries the same rank with **no gaps**, so rank 2 is the true second-*distinct* salary. `RANK` would skip to 3 after a tie, and `ROW_NUMBER` would treat two equal top earners as ranks 1 and 2 — returning a duplicate of the highest, which is wrong.

The tempting one-liner to avoid: `ORDER BY salary DESC LIMIT 1 OFFSET 1`. It's clean but **breaks with duplicate top salaries** — if two people share the highest salary, OFFSET 1 hands you that top value again, not the second-highest distinct salary. It only works when you can guarantee no ties. So I'd write the subquery or `DENSE_RANK` and explicitly call out the duplicate edge case — that's what separates a correct answer from a lucky one.
