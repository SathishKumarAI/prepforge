I don't guess — I measure. First step is always **`EXPLAIN ANALYZE`** to read the execution plan and find the actual bottleneck: a full table scan where an index should kick in, an expensive sort, a nested-loop join blowing up on large inputs, or a bad row-count estimate. The plan tells you *where* the time goes; optimizing anything else is superstition.

Then, targeted fixes:

- **Indexing** — add indexes on columns in `WHERE`, `JOIN`, `ORDER BY`, `GROUP BY`. Use **composite indexes** whose column order matches the query's predicates, and **covering indexes** so the query is answered from the index alone. But don't over-index — every index taxes writes.
- **Keep predicates sargable** — the highest-impact and most-missed fix. `WHERE YEAR(order_date) = 2024` wraps the column in a function and **kills index usage**; rewrite as `WHERE order_date >= '2024-01-01' AND order_date < '2025-01-01'`. Same result, index now usable.
- **Select only needed columns** — `SELECT *` drags unnecessary I/O and defeats covering indexes.
- **Filter and aggregate early**, before joins, to shrink the rows flowing through the plan; join on indexed keys.
- **Structural options** for big data — **partitioning** (e.g., by date) to prune whole segments, and **materialized views** to precompute expensive aggregations.
- **Refresh optimizer statistics** so it stops picking bad plans.

And crucially, **benchmark before and after** on realistic data. I've watched "obvious" index additions do nothing because the real culprit was a non-sargable predicate or stale stats. Let the execution plan, not intuition, drive every change.
