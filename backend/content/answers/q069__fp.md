The root issue: most models are functions over ℝⁿ — they compute distances, dot products, gradients. A category like `{red, green, blue}` has no arithmetic. So encoding is the act of *embedding an unordered set into a vector space* without smuggling in false structure. Every method is a different answer to "what geometry does this category actually have?"

- **One-hot encoding:** map each level to an orthogonal basis vector. This asserts *no order and equal distance* between categories — the honest default for nominal variables. Cost: dimensionality explodes with cardinality (the geometry gets sparse and high-dimensional).
- **Ordinal/label encoding:** map to integers. Valid *only* when a true order exists (low<med<high); using it on nominal data injects a fake distance metric the model will believe.
- **Target/mean encoding:** replace a level with the mean target for that level. This projects the category directly onto the axis that matters — powerful for high cardinality, but it *leaks the label*, so it must be computed with cross-fold/smoothing or it memorizes noise.
- **Learned embeddings:** let the model place categories in a dense low-dim space by gradient descent, discovering that "ZIP 10001" sits near "10002."

The insight most miss: encoding is a *modeling assumption about similarity*, not preprocessing. Tree models split on any encoding and barely care about one-hot vs. ordinal; distance/linear models care enormously — the encoding *defines* the space they reason in.
