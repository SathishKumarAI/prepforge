question: "What does GROUP BY do in SQL, and how does HAVING differ from WHERE?"

`GROUP BY` performs a fundamental transformation: it **collapses many rows into one row per group**, partitioning the table by the grouping key and reducing each partition to a single summary via aggregate functions (COUNT, SUM, AVG). Conceptually it's a fold — it changes the *grain* of the data from "one row per event" to "one row per category." Everything in the SELECT must then be either a grouping key (constant within a group) or an aggregate (a reduction over the group); anything else is undefined, because the group has no single value for it.

The WHERE-vs-HAVING distinction falls straight out of *when* each filter runs relative to that collapse:

- **WHERE** filters **individual rows before grouping.** It operates on the raw grain, so it cannot see aggregates — they don't exist yet. It decides which rows even enter the buckets.
- **HAVING** filters **groups after aggregation.** It operates on the collapsed grain, so it *can* reference aggregates like `SUM(x) > 1000`.

The principle is pipeline ordering: SQL logically executes FROM → WHERE → GROUP BY → HAVING → SELECT. A clause can only reference what already exists at its stage.

The insight most miss: this ordering is also a *performance* lesson. Push conditions into WHERE whenever possible — filtering rows *before* grouping shrinks the work; forcing a row-level condition into HAVING makes the engine aggregate data it will only throw away.
