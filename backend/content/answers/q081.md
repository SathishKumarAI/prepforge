A **data warehouse** is a system optimized for **analytical queries over large volumes of historical, integrated data** — think Snowflake, BigQuery, Redshift, or Databricks. A "database" in the usual sense means an **OLTP** system (Postgres, MySQL) optimized for **transactions**: many small, concurrent reads and writes serving an application.

The core difference is the workload, and it drives everything else:

- **OLTP** is row-oriented, heavily normalized (3NF), tuned for fast single-record inserts/updates with strict consistency. It answers "what is user 12345's current balance?"
- **OLAP / warehouse** is columnar, often denormalized into star schemas, tuned to scan and aggregate billions of rows. It answers "what was revenue by region by month for three years?"

Columnar storage is the big lever — scanning three columns out of two hundred touches only those columns, and compression on homogeneous column data is dramatic. Warehouses also separate storage from compute (in the cloud generation), so you scale query power independently.

In production you almost never run heavy analytics against the production OLTP database — it competes with live traffic and locks tables. You **ETL/ELT** data out into the warehouse. A related term, **data lake**, stores raw files (Parquet/JSON) cheaply; the "lakehouse" pattern (Databricks, Iceberg) blurs the line by putting warehouse-style query engines over lake storage.
