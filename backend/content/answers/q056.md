The distinction is about how rigidly the data conforms to a predefined schema. **Structured** data lives in a fixed schema — rows and columns with typed fields — like a SQL table or a clean CSV. **Semi-structured** data has organizational markers (tags, keys, nesting) but no rigid table schema: JSON, XML, YAML, log lines, Parquet with nested fields. **Unstructured** data has no inherent field structure at all: free text, images, audio, video, PDFs.

The practical consequence is *how you query and store it*. Structured data is the home turf of relational databases and SQL. Semi-structured fits document stores (MongoDB), or columnar formats and query engines that handle nesting (BigQuery, Spark, DuckDB over JSON/Parquet). Unstructured data usually lands in object storage (S3) and only becomes queryable after you extract features or embeddings from it.

Estimates put roughly 80% of enterprise data in the unstructured bucket, and that's exactly where the modern value — and difficulty — sits. In production the line blurs: a "structured" events table often has a `properties` JSON blob that's really semi-structured, and teams get burned assuming they can filter on keys inside it at scale without indexing. The modeling implication: structured data suits classical ML (trees, regression) directly, while unstructured data needs deep learning or embeddings to turn into something a model can consume.
