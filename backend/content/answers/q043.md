Feature scaling puts numeric features on a comparable range so that no single feature dominates purely because of its units. The two workhorses are **standardization** (z-score: subtract mean, divide by std → mean 0, unit variance) and **min-max normalization** (rescale to `[0,1]`). Use standardization by default; min-max when you need a bounded range (e.g., image pixels) but watch that it's sensitive to outliers. For heavy-tailed data, `RobustScaler` (median and IQR) survives outliers better.

Which algorithms need it? Anything that depends on **distances, dot products, or gradient geometry**:

- **Distance-based**: k-NN, k-means, SVM with RBF/linear kernels, PCA — all dominated by large-range features if unscaled.
- **Gradient descent** models: linear/logistic regression and neural nets converge far faster on scaled inputs; unscaled features give elongated loss surfaces that zig-zag.
- **Regularized** models (Ridge, Lasso): the penalty is applied per-coefficient, so unscaled features get unfairly penalized.

Which don't? **Tree-based** models — decision trees, Random Forest, XGBoost, LightGBM — are invariant to monotonic feature transforms because they split on thresholds, so scaling is a no-op there.

The discipline that matters in production: **fit the scaler on training data only**, then apply those same statistics to validation, test, and live traffic. Fitting on the full dataset leaks test-set statistics; forgetting to persist and reapply the scaler at serving time is a classic train/serve skew bug. Wrap it in a pipeline.
