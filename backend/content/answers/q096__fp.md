Reason from the physics: a query is slow because it makes the machine *touch too much data* or touch it *inefficiently*. Every optimization is an attack on one of those. So the first move is never to guess—it's to **read the execution plan** (`EXPLAIN ANALYZE`), because the plan tells you exactly where rows are being read, joined, and thrown away.

The dominant lever is **avoiding full scans on selective filters**. If `WHERE user_id = 42` scans a billion rows to keep ten, you're paying O(N) for an O(log N) job. An index turns that linear scan into a tree descent—the single biggest win, and it's really about *reducing the cardinality processed as early as possible*.

From that principle the rest follows:
- **Filter early, project narrow** — push predicates down, select only needed columns.
- **Fix join order/type** — a nested-loop join over unindexed keys is quadratic; the right index makes it a hash or merge join.
- **Kill hidden killers** — functions on indexed columns (`WHERE DATE(ts)=…`) defeat the index; `SELECT *` and correlated subqueries force rescans.
- **Precompute** when the same aggregate is read repeatedly (materialized views).

The non-obvious insight: the goal isn't "use indexes," it's **maximize data eliminated per unit of work, as early as possible.** On a query returning *most* of the table, an index is *slower* than a scan—optimization is always relative to selectivity, never absolute.
