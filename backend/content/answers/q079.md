Joins combine rows from two tables on a matching key; the four types differ in **what happens to rows that don't match**.

- **INNER JOIN** — only rows with a match in **both** tables. Non-matches from either side vanish.
- **LEFT (OUTER) JOIN** — **all** left-table rows, plus matches from the right; where there's no match, right-side columns come back **NULL**.
- **RIGHT (OUTER) JOIN** — the mirror: all right-table rows, NULLs for missing left matches. (I rarely write these — I just flip the table order and use LEFT, which reads more naturally.)
- **FULL OUTER JOIN** — all rows from **both** sides, NULLs wherever either side has no match.

```sql
SELECT c.name, o.id
FROM customers c
LEFT JOIN orders o ON c.id = o.customer_id;
```

This lists every customer *including those with zero orders* (their `o.id` is NULL) — exactly what you want for "who hasn't purchased?"

Two things that bite people in real analysis. First, the **anti-join** pattern: `LEFT JOIN ... WHERE right.key IS NULL` finds left rows with no counterpart — customers with no orders, users with no events. Second, and more dangerous, **fan-out**: if the right table has multiple matches per key, a join *multiplies* rows and silently inflates any `SUM` or `COUNT` downstream. I've debugged plenty of "revenue is suddenly 3x" incidents that traced to a one-to-many join no one accounted for. So I always sanity-check row counts before and after a join.
