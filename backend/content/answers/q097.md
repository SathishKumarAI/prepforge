Retention asks: of the users who first appeared in a given month, what fraction came back N months later? So the pattern is always **define a cohort by first-activity month, then self-join that cohort against later activity and count the overlap.**

Two building blocks:

1. A **cohort table** — each user's first active month.
2. An **activity table** — the distinct months each user was active.

```sql
WITH cohort AS (
  SELECT user_id,
         DATE_TRUNC('month', MIN(event_date)) AS cohort_month
  FROM events
  GROUP BY user_id
),
activity AS (
  SELECT DISTINCT user_id,
         DATE_TRUNC('month', event_date) AS active_month
  FROM events
)
SELECT c.cohort_month,
       DATE_DIFF('month', c.cohort_month, a.active_month) AS month_number,
       COUNT(DISTINCT a.user_id)                          AS retained_users
FROM cohort c
JOIN activity a ON c.user_id = a.user_id
GROUP BY 1, 2
ORDER BY 1, 2;
```

`month_number = 0` is the cohort size (everyone is active their first month); dividing each row by that baseline gives the retention **percentage**, producing the classic retention triangle.

Details that bite in production: pin `event_date` to the user's timezone before truncating, use `COUNT(DISTINCT user_id)` to avoid double-counting multi-event months, and decide upfront whether you want **classic retention** (active in exactly month N) or **rolling/unbounded retention** (active in month N *or later*) — they answer different questions and stakeholders assume different ones. `DATE_TRUNC`/`DATE_DIFF` syntax varies by warehouse but the shape is identical.
