question: "What is the difference between a primary key and a foreign key?"
topic: "Data Analytics"

A **primary key** uniquely identifies each row in a table; a **foreign key** is a column in one table that references the primary key of another, creating the relationship between them.

A primary key must be **unique and non-null** — no two rows share it, and it's never empty. It's the anchor a row is looked up and joined by. A foreign key enforces **referential integrity**: an `orders.customer_id` foreign key pointing at `customers.id` guarantees you can't insert an order for a customer that doesn't exist, and (depending on the constraint) controls what happens on delete — `CASCADE`, `SET NULL`, or `RESTRICT`.

Two nuances worth stating in an interview:

- A foreign key value is **not** required to be unique — many orders belong to one customer (the "many" side of one-to-many). It also can be null (an order not yet assigned).
- Primary keys can be **natural** (a real attribute like an ISO country code) or **surrogate** (a meaningless auto-increment or UUID). In production I almost always favor surrogate keys — natural keys have a habit of changing (emails, SSNs get corrected) and cascading that change through every referencing table is painful.

In analytical warehouses, be aware that engines like BigQuery and Redshift often **don't enforce** these constraints — they're declared for documentation and query-planner hints, so data quality checks must catch orphaned foreign keys yourself.
