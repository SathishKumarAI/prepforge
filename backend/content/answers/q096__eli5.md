question: "How would you optimize a slow SQL query?"

Speeding up a slow query starts with looking, not guessing. Run **EXPLAIN** to see the query's plan, the database tells you where it's struggling, like scanning an entire table row by row.

The biggest win is usually an **index**. Imagine finding every mention of a word in a book: without an index you read every page; with one you flip to the listing at the back. So add indexes on the columns you filter, join, and sort on. But don't wrap those columns in functions (like `YEAR(date)`), that blinds the index.

Other fixes: select only the columns you actually need instead of `SELECT *`, filter early to shrink the data before joining, join on indexed keys, and limit big results with pagination. Keep the database's internal statistics up to date so it picks a smart plan.

Above all, measure before and after, and let the plan, not a hunch, guide each change.
