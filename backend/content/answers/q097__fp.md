Retention is fundamentally a **set-membership question across time**: of the users who existed at time T, what fraction *reappear* at time T+k? So the whole computation is a self-comparison of a table against a time-shifted version of itself. Everything else is bookkeeping.

Decompose it into the three things you must define:

1. **The cohort (the denominator).** Anchor each user to an origin—typically their first-active month. `MIN(activity_month)` per user gives the "birth month." This is the fixed reference population; without it you're measuring a churning crowd, not retention.
2. **The return event (the numerator).** For each cohort, find which of its users are active in month +1, +2, …. Conceptually a self-join: `cohort_users` ⋈ `activity` on `user_id`, keeping rows where the activity month is *k months after* the cohort month.
3. **The ratio.** `returning_users / cohort_size`, grouped by cohort month and offset—producing the familiar triangular retention grid.

The core mechanic is that self-join (or a window function like `LAG`/`DATEDIFF` on activity months) that lets one user's *later* behavior be measured against their *own* origin.

The non-obvious insight most miss: retention is **defined by what counts as "active."** Distinct-login vs. any-event vs. a meaningful action produces wildly different curves from identical data. The SQL is trivial; the semantic choice of the return event *is* the analysis, and it must be fixed before a single number is trustworthy.
