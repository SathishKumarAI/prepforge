Data leakage is when information that *wouldn't be available at prediction time* — or the target itself, in disguise — sneaks into training. The result is a model that looks brilliant in validation and collapses in production. It's the most common and most dangerous mistake I see, precisely because it hides behind *great* metrics, so nobody goes looking.

The forms I actively hunt for:

- **Target leakage** — a feature is a proxy for the label. Classic: using `account_closed_date` to predict churn, or including a "risk_flag" that was set *after* the outcome. The feature is basically the answer.
- **Preprocessing contamination** — fitting a scaler, imputer, encoder, or feature selector on the *full* dataset before splitting. Now test-set statistics have leaked into training. This one is subtle and everywhere.
- **Temporal leakage** — using future rows to predict the past in time-series, or a random shuffle that scatters future data into the train fold.
- **Duplicate/near-duplicate rows** straddling train and test (same user, multiple sessions).

Prevention is disciplined, not clever: **split first**, then fit *all* transforms on the training fold only — enforce this with a scikit-learn `Pipeline` so every step is refit per CV fold. Respect **time order** with time-based splits for temporal data. Audit each feature by asking "would I actually have this value at the moment of prediction?" And treat a **too-good-to-be-true metric as a bug report, not a win** — 0.99 AUC on a hard problem means find the leak before you celebrate.
