question: "What does GROUP BY do in SQL, and how does HAVING differ from WHERE?"

`GROUP BY` collapses rows sharing the same key values into one row per group, so you can apply **aggregate functions** — `COUNT`, `SUM`, `AVG`, `MIN`, `MAX` — per group instead of over the whole table.

```sql
SELECT department, COUNT(*) AS headcount, AVG(salary) AS avg_salary
FROM employees
GROUP BY department;
```

`WHERE` vs `HAVING` comes down to **when they filter relative to aggregation**. `WHERE` filters **individual rows before** grouping and cannot reference aggregates. `HAVING` filters **groups after** aggregation and *can* use aggregate functions. Use both together and each does its job:

```sql
SELECT department, AVG(salary) AS avg_salary
FROM employees
WHERE active = 1                 -- drop inactive rows first
GROUP BY department
HAVING AVG(salary) > 50000;      -- then keep high-paying departments
```

The reason this ordering isn't arbitrary is SQL's **logical execution order**: `FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY`. Two consequences fall right out of it, and both are classic interview traps. You **can't reference a SELECT alias in WHERE** (SELECT runs later), and you **can't put an aggregate in WHERE** (`WHERE COUNT(*) > 5` errors — it must be HAVING, because aggregates don't exist until after GROUP BY).

The performance note I'd add: filter as much as possible in `WHERE`, not `HAVING`. `WHERE` shrinks the row set *before* the engine does the expensive grouping work, so pushing a non-aggregate condition into `WHERE` instead of `HAVING` is usually a free speedup.
