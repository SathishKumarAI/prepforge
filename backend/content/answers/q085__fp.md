Ask the foundational question: *when can data be trusted to represent reality?* Every quality dimension is a distinct way the map can diverge from the territory, and enumerating the failure modes derives the list.

- **Accuracy** — does a value match the real-world fact? (Address is genuinely where the customer lives.)
- **Completeness** — are required values present, or riddled with nulls? Missingness silently biases every aggregate.
- **Consistency** — do copies agree? The same customer's status shouldn't differ across two tables.
- **Uniqueness** — is each real entity represented once? Duplicates inflate counts and double-charge.
- **Timeliness** — is the data current enough for the decision? A correct value that's a week stale can be functionally wrong.
- **Validity** — does it conform to defined rules/formats? (A date that parses, an enum in range.)

The unifying principle is **entropy of a communication channel**. Reality is the source; your database is the received signal after a noisy pipeline of sensors, humans, and joins. Each dimension measures a different corruption the channel introduces.

The non-obvious insight: these dimensions **trade off**. Waiting to verify accuracy costs timeliness; deduplicating aggressively risks discarding genuinely distinct records. "Data quality" isn't a single knob to maximize—it's a *fitness-for-purpose* judgment, defined only relative to the decision the data must support.
