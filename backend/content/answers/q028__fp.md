question: "Why do we split data into training, validation, and test sets?"

The core problem is that we want to estimate **future performance** but only have past data. Any performance number computed on data the model has *touched* is optimistically biased — the model can exploit that specific sample. So we must reserve data the model has never influenced, to get an honest estimate. That's the whole justification for splitting.

But there are two *distinct* ways a model touches data, which is why we need *three* sets, not two:

- **Training set** — the model fits its **parameters** here. Error here is hopelessly optimistic.
- **Validation set** — used to choose **hyperparameters** and model form (architecture, regularization, early stopping). Crucially, *you* are now optimizing against it by trial and error. After many choices, the validation score also becomes optimistic — you've indirectly fit to it.
- **Test set** — untouched by *both* fitting and model-selection, used *once* at the end for the final unbiased estimate.

The unifying principle: **every decision informed by a dataset "uses up" that set's ability to give an unbiased estimate.** Each level of decision-making needs its own fresh holdout.

The non-obvious insight: the test set is a *one-shot resource*. Peek at it repeatedly to guide choices and it silently degrades into a second validation set — this "leakage through the researcher" is why leaderboards overfit and reported numbers don't survive deployment.
