In a zero-sum game your gain is exactly the opponent's loss. So to choose well you cannot assume the opponent cooperates — you must assume they play *against* you. **Minimax** encodes this pessimism: label your moves MAX (maximize the score) and the opponent's MIN (minimize it), then recurse to the leaves where the outcome is known, and propagate values back up. At each node you compute the value under the assumption that both sides play optimally. The chosen move is the one whose worst-case outcome is best — a saddle point of the game tree.

The problem: the tree is b^d in size (branching factor to the power of depth). Exhaustive search is hopeless for chess. **Alpha-beta pruning** exploits a logical shortcut: track α (best score MAX can already guarantee) and β (best MIN can guarantee). If exploring a branch reveals a value proving that a rational opponent would never let you reach this node — the branch can't affect the final decision — you *prune* it unexplored. You're skipping computations whose outcome cannot change the answer.

The non-obvious insight: pruning changes nothing about the *result* — it returns the identical minimax value — only the *cost*. With perfect move ordering it cuts the effective branching factor to √b, roughly *doubling* searchable depth for the same work. Search efficiency, not different logic.
