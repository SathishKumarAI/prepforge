The right encoding depends on cardinality, whether the categories are ordinal, and the model you're feeding.

**One-hot encoding** is the safe default for low-cardinality nominal features (color, country-of-a-handful): one binary column per category, no false ordering imposed. The cost is dimensionality — encode a 10,000-value ID column this way and you've blown up your feature space and memory.

**Ordinal/label encoding** maps categories to integers. It's correct only when there's a genuine order (small < medium < large). Applying it to nominal data (`red=1, blue=2, green=3`) injects a fake ordinality that linear and distance models will happily misuse — though tree models tolerate it, since they can split arbitrarily.

For **high-cardinality** features (zip code, product ID, user agent), the real tools are:

- **Target/mean encoding**: replace each category with a (smoothed) statistic of the target. Powerful, but it *leaks* target information — you must compute it inside cross-validation folds or with out-of-fold encoding, and add smoothing/noise, or you'll overfit spectacularly. This is the most common footgun I clean up in production.
- **Frequency/count encoding**: replace with how often the category appears — cheap and often surprisingly effective.
- **Embeddings**: a learned dense vector per category, standard in neural nets and great when categories have latent structure.

Practical notes: fit encoders on training data only, and decide up front how you'll handle **unseen categories** at inference — an unmapped value shouldn't crash the pipeline. Gradient-boosting libraries like CatBoost and LightGBM handle categoricals natively and often beat hand-rolled encodings.
