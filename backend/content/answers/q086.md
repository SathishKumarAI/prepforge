Window functions compute across a set of rows related to the current row — a "window" — **without collapsing** them. That's the whole difference from `GROUP BY`: GROUP BY reduces each group to one summary row; a window function adds a computed column while **keeping every original row**.

```sql
SELECT name, department, salary,
       AVG(salary)  OVER (PARTITION BY department)               AS dept_avg,
       RANK()       OVER (PARTITION BY department ORDER BY salary DESC) AS rnk
FROM employees;
```

The mechanics live in `OVER()`: **PARTITION BY** groups rows (like GROUP BY, but detail survives), **ORDER BY** sequences them within the partition (needed for ranking and running totals), and an optional **frame** (`ROWS BETWEEN ...`) defines the sliding subset for moving calculations.

Where they're indispensable in analytics — things that are painful or impossible with GROUP BY alone:

- **Ranking / top-N per group**: `ROW_NUMBER`, `RANK`, `DENSE_RANK` — e.g., the 3 highest-value orders *per customer*.
- **Period-over-period**: `LAG`/`LEAD` to compare each row to the previous month without a self-join.
- **Running totals and moving averages** via a frame.
- **Comparing a row to its group's aggregate** — "this sale vs. the department average" — in one pass, keeping the row.

The move I use constantly: wrap a `ROW_NUMBER() OVER (PARTITION BY ... ORDER BY ...)` in a subquery and filter `WHERE rn = 1` to **deduplicate** — grab the latest record per entity. And the subtle interview point: `RANK` leaves gaps after ties (1,1,3), `DENSE_RANK` doesn't (1,1,2), and `ROW_NUMBER` breaks ties arbitrarily — picking the wrong one is a common bug.
