A knowledge graph represents information as a network of **entities** (nodes) connected by typed **relationships** (edges), with facts stored as triples: `(subject, predicate, object)` — e.g., `(Paris, capitalOf, France)`. The value over a flat table is that meaning lives in the connections: you can traverse relationships, follow transitive links, and run inference the raw data never stated explicitly.

Under the hood you'll see two flavors: RDF/SPARQL with formal ontologies (OWL, schema.org) for standardized, reasoning-heavy domains, and property graphs (Neo4j, Cypher) that attach key-value properties directly to nodes and edges, which most application teams find more ergonomic. An ontology or schema layer defines valid entity types and relations and is what lets you catch contradictions and derive new facts.

They power Google's Knowledge Graph, product/recommendation catalogs, fraud rings (graph structure exposes them cleanly), and increasingly **grounding for LLMs** — GraphRAG-style retrieval where you pull a connected subgraph as context instead of loose text chunks, which improves multi-hop questions.

The honest production caveat: the hard part isn't the graph, it's entity resolution and keeping it current. Deduplicating "IBM" vs "International Business Machines," reconciling messy source systems, and maintaining freshness is where most of the effort goes. A stale or poorly-linked knowledge graph gives you confident, wrong traversals. Build the ingestion/resolution pipeline first; the query layer is the easy part.
