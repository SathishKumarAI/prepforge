question: "What is database normalization and what problem does it solve?"

Normalization organizes a relational schema so **each fact is stored exactly once**, splitting data into related tables to cut redundancy and protect integrity. The problem it solves is a trio of **anomalies** that redundancy creates:

- **Update anomaly** — a fact duplicated across many rows must be changed everywhere; miss one and the data now contradicts itself.
- **Insertion anomaly** — you can't record one thing without unrelated data present (can't add a product with no order to attach it to).
- **Deletion anomaly** — removing a row accidentally destroys other facts stored only there.

The normal forms are progressive rules: **1NF** — atomic values, no repeating groups; **2NF** — no partial dependency on part of a composite key; **3NF** — no transitive dependencies, so every non-key column depends on the key, the whole key, and nothing but the key. 3NF is the practical target for transactional systems.

Here's the tension every analytics engineer has to hold, and what makes this a good interview question: normalization is **ideal for OLTP** — write integrity, no contradictions, the source of truth. But it's the *wrong* default for **analytics**, because reconstructing anything meaningful requires many joins, which get slow and unwieldy over large historical tables. So warehouses deliberately go the other way and **denormalize** into star schemas, trading storage redundancy for query speed and simplicity.

The clean framing: **normalize for integrity, denormalize for read performance.** Neither is "correct" universally — you pick based on whether the system is optimized for consistent writes (normalize) or fast analytical reads (denormalize).
