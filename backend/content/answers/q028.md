question: "Why do we split data into training, validation, and test sets?"
topic: "Machine Learning"
model: "Claude Code (authored)"
sources: []
---

Because you need an honest estimate of how the model behaves on data it has never seen, and you can only get that if some data stays genuinely untouched. Train fits the parameters, validation tunes the choices you make (hyperparameters, architecture, features, early stopping), and test is the final, one-time report card you touch once at the end.

The subtle point is that the validation set gets *contaminated* by the act of tuning. Every time you pick a model because it scored better on validation, you leak a little of that set into your decisions, so validation performance drifts optimistic. The test set exists precisely to catch that. If you peek at test repeatedly and tweak, you've turned it into a second validation set and lost your unbiased estimate — one of the most common ways teams ship a model that looks great offline and disappoints in production.

Typical splits are 60/20/20 or 70/15/15; with little data, use k-fold cross-validation instead of a fixed validation set. Two things I insist on: split *before* any preprocessing so scalers and encoders are fit on train only (otherwise you leak test statistics), and split by the right unit — by user or by time, not by row — when observations are correlated or the real task is forecasting the future.
