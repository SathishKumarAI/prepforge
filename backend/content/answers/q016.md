Hallucination is when a model produces fluent, confident output that's factually wrong or fabricated — invented citations, made-up API methods, plausible-but-false dates. The root cause is structural, not a bug: the model is trained to predict the *most likely next token*, and a well-formed fabrication is often more probable than an honest "I don't know." It optimizes plausibility, not truth, and it has no built-in mechanism to check itself.

Because it's structural, you manage it rather than eliminate it. The single highest-leverage fix is **RAG** — ground answers in retrieved sources so the model quotes evidence instead of recalling from fuzzy weights, and it can cite. Second, prompt design: explicitly permit "I don't know," ask it to answer *only* from provided context, and request citations you can verify. Third, **lower temperature** for factual tasks so it stops sampling into the creative tail. Fourth, **verification layers** — validate generated code by running it, check that cited sources exist, use a second model or rules as a fact-checker, and constrain outputs to a schema.

Two hard-won lessons from production. Grounding is only as good as retrieval — if the right document isn't retrieved, the model hallucinates anyway, so measure retrieval quality separately. And there's no threshold that makes it safe for unattended high-stakes use: for anything consequential — medical, legal, financial — keep a human in the loop and treat confident fluency as *no evidence at all* of correctness.
