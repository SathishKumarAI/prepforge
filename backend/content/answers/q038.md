These come straight off the confusion matrix and exist because accuracy lies on imbalanced data. **Precision = TP/(TP+FP)** — of everything you flagged positive, how much was right; it punishes **false positives**. **Recall = TP/(TP+FN)** — of all true positives out there, how many you caught; it punishes **false negatives**. **F1 = 2·(P·R)/(P+R)**, the harmonic mean, gives a single balanced number and, unlike accuracy, stays honest when classes are skewed.

Which to prioritize is a *business* decision about which error hurts more:

- **Optimize recall** when a miss is catastrophic — cancer screening, fraud, security threats. You'd rather tolerate false alarms than let a real case through.
- **Optimize precision** when a false alarm is costly or erodes trust — spam filtering (don't trash a real email), or triggering an expensive/irreversible intervention.
- **Use F1** when you need one metric and both errors matter, especially with class imbalance.

The point I always make: there's a **precision–recall trade-off** governed by the decision threshold, so quoting either number without the threshold is meaningless. Lower the threshold and recall rises while precision falls. So in practice I don't chase a single point — I look at the **precision–recall curve**, pick the operating point that matches the cost of each error, and set the threshold there. On heavily imbalanced problems I report **PR-AUC** rather than ROC-AUC, because ROC can look flatteringly good when true negatives dominate.
